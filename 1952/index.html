[Discussion] A class should have state or dependencies, but not both

May 3, 2021 by usernameqwerty005

This principle “feels” right, but I’m working through trying to formulate it properly.

Consider a point class (in pseudo-code because lazy):

class Point
private x, y
construct(x, y)
draw(Drawable surface)

(Instead of surface, it could also be a save-function with a database connection as argument (or indeed any type of persistence)).

If the surface was instead injected into the constructor, the point would look like:

class Point
private x, y, surface
constructor(x, y, surface)

Isn’t that weird, to have two properties related to state, and then a third property related to something completely different? Especially when it’s something effectful, like drawing or writing to disk.

On the other hand, you can imagine a command object class like this:

class InstallApp
private db, io, logger, mailer
constructor(db, io, logger, mailer)
execute(app)

In this case, _all_ dependencies are effectful classes. Makes more sense, right? And then the app class has only “pure” properties, like `User` or `Configuration`.

A rectangle depending on points is also OK, since the points are pure:

class Rectangle
private bottomLeft, topRight
constructor(Point bottomLeft, Point topRight)
// draw, save, etc

Another way to phrase it is that classes should only depend on other classes in the same layer (domain layer vs “effectful” layer).

The major drawback is that no language can actually distinguish between a dependency and “normal” class property. Possibly they should have different semantics? Or the possibility to separate pure and effectful classes.

Thoughts?

  • umulmrum says:
    at

    I have the same “feeling” as OP, but wasn’t completely able to express the reasons myself yet. Let me try a few statements:

    State complicates things. If there is a problem in your code, it’s a lot easier to spot the cause if there is either no state or only local state involved. So I prefer to have state only on stack objects such as entities, commands, DTOs and the like; all of these are easy to instantiate if the new operator can be used which is the case if only state variables are passed to the constructor and no dependencies. Otherwise we need to get these dependencies from somewhere, either directly in the caller (which most likely leads to code duplication if there are multiple places where that object is instantiated; also this might qualify for the Feature Envy code smell) or in some kind of factory.

    Whenever I can I use the Symfony DI container, which is why I think of code with dependencies as services in a static (= dumped to file in a CI pipeline) environment, so no dynamic state is possible unless a factory is used. My thinking is that the container forms a frame that is not to be changed during a request. Maybe this is because I coded in Java in the beginning of my career where the booted application stays in memory and is not wiped after each request, so that the application structure that does not depend on mutable data must stay the same during a request.

    So from my perspective an object should either be “easily” created dynamically or injected by the DIC. Hybrid forms add a layer of complexity. They also require that to some degree dependency management is done in application code where I don’t want to have it. Other advantages:

    – A class without dependencies doesn’t have side effects (unless you do weird things with global state) which again makes it easier to understand and to test.

    – An object without dependencies can easily be serialized to any format, as deserialization does not require restoring the same dependencies.

    &#x200B;

    That said I think in some cases hybrid classes do make sense. As always, no rule without exception 🙂

  • Crell says:
    at

    It becomes a lot easier to describe what you’re getting at when you stop thinking of classes and instead think of product types (the combination of two or more other types, aka value object) and closures (a dynamic function that has some of its parameters pre-filled), both of which happen to share the same syntax.

    A do-object is a closure with funny syntax. Any given method of a service object is essentially the same as a single function that takes all of the constructor parameters itself, which you’ve partially applied to make a closure. That is:

    // This:
    class DoThing {
    public function __construct(private Dependency $dep) {}

    public function first(int $a, string $b): string { … }

    public function second(string $c): string { … }
    }

    // Is effectively the same as this:

    function doFirst(Dependency $dep, int $a, string $b): string { … }
    function doSecond(Dependency $dep, string $c): string { … }

    $dep = get_dependency_from_somewhere();
    $first = fn(int $a, string $b) => doFirst($dep, $a, $b);
    $second = fn(string $c) => doSecond($dep, $b);

    (The syntax for the second version is likely going to get even nicer in PHP 8.1, assuming the partial application RFC passes.)

    And yes, it shouldn’t carry any state, just partially applied values. Those make no sense to ever change.

    Conversely, a thing-object represents a value. A Point is a value, a thing, composed of two other things (ints, for example). It has no behavior, other than short-hand convenience operations on it. Say, areaUnder(). But really, that’s also the same thing.

    class Point {
    public function __construct(public int $x, public int $y) {}

    public function areaUnder(): int { … }
    }

    // vs.
    class Point {
    public function __construct(public int $x, public int $y) {}
    }

    function areaUnder(Point $p): int { … }

    Those are logically equivalent; the former is just a convenient sugar for the latter.

    So Point is, or has, state. However, it’s better for Point to *be* state, not to *have* state, because the Point 3, 5 is the same as any other Point 3, 5 but is not at all the same as the Point 3, 7. So you really don’t want Point to *have* any mutable state. It is itself a fixed value, and a different point is… a different point.

    So a guideline of “a class should have dependencies OR it should have values” (not quite the same as state) is a good one, because it is effectively wrapping up the distinction between product types and closures.

    For more of that kind of thinking and insight, have a gander at my book on the topic, “Thinking Functionally in PHP.” 🙂

    [https://leanpub.com/thinking-functionally-in-php](https://leanpub.com/thinking-functionally-in-php)
